ANBS peer to peer connection module wraps host-provided WebRTC primitives to create transportable offer and contract values, establish a data channel between two peers, exchange structured application messages, and optionally share microphone, camera, and screen media.

This specification is maintained alongside a reference implementation.

Introduction

The module provides an abstraction over peer connections and data channels as defined by [[WEBRTC]]. Instead of bundling a signaling transport, it produces serializable values that callers move through their own signaling channel, such as WebSocket, HTTP, QR, copy-paste, or any other transport.

After two peers exchange an offer and the resulting contract copies, each side constructs a live connection abstraction from one contract copy. The connection abstraction exposes operations for readiness, application messages,media publishing, and remote media observation . Media capture is built on user-media facilities from [[MEDIACAPTURE_STREAMS]] and display-capture facilities from [[SCREEN_CAPTURE]].

This specification does not define any wire protocol for moving offers, answers, or contract copies between peers. The application or embedding system is responsible for that transport.

Conformance

A conforming implementation of an ANBS peer to peer connection module MUST expose the concepts, operations, values, and observable behavior defined in this document in a form natural to the host language and runtime. Unless stated otherwise, lists, strings, maps, and similar structural terms are to be interpreted consistently with [[INFRA]].

Operation names such as makeOffer, acceptOffer, ready, and sendMessage identify abstract package operations. A binding MAY expose those operations as methods, functions, procedures, traits, services, modules, or any equivalent host-language construct, provided the specified behavior is preserved.

Data Records

Offer

An Offer record transports the local session description generated by one peer.

Member Type Meaning
offerId Text string Correlation identifier generated using a host UUID facility.
description Host WebRTC session-description value Local SDP offer after ICE gathering has completed.

Contract Copies

Accepting an offer produces two contract copies. One copy returns to the original offer creator, and the other remains on the accepting peer.

Record Members Meaning
OfferorCopy offerId, role, answer Returned to the original offer creator. The role member MUST equal "offeror".
OffereeCopy offerId, role Retained by the accepting peer. The role member MUST equal "offeree".
Contract Union of OfferorCopy and OffereeCopy Input to the connection-construction operation.
ContractCopies offeror, offeree Pair returned by acceptOffer().

Reserved Internal Signal

The package reserves the following message shape for internal renegotiation traffic carried over the data channel: 

{
  __sovereignbase_peer2peer: 'renegotiate-offer' | 'renegotiate-answer',
  description: SessionDescriptionInit
}

Application payloads that match this shape are consumed internally and are not surfaced as ordinary application-message events.

Error Model

Package-defined guard failures use P2PConnectionError. The abstraction exposes a stable code value whose value is one of the P2PConnectionErrorCode literals.

A P2PConnectionError message MUST be prefixed with the package scope string {@sovereignbase/peer2peer}. When a specific message is not supplied, the code itself is used as the error detail.

A binding MAY expose this failure abstraction as an exception type, an error object, a tagged result variant, or any equivalent host-language error mechanism, provided the stable code and message behavior are preserved.

Code Meaning
CHANNEL_ERROR The data channel emitted an error indication before reaching the "open" state.
CHANNEL_CLOSED The data channel closed before reaching the "open" state.
CHANNEL_NOT_AVAILABLE The peer connection reached "failed" or "closed" before an incoming data channel was received.
CONNECTION_NOT_READY sendMessage() was invoked before the local data channel was available and open.
UNKNOWN_PEER_CONTRACT The connection-construction operation received a contract whose reserved local state had not been created or had already been consumed.
MISSING_LOCAL_DESCRIPTION An offer or answer could not be read from the host peer connection's local-description slot after ICE gathering completed.

Host-environment failures, such as offer creation, remote description application, user-media capture, or display capture rejections, are not wrapped by the package and may propagate as host-defined exceptions or failure values.

Event Model

Each connection instance defines a connection-scoped notification surface. The package-defined event kinds are listed below.

Event kind Payload When emitted
message Decoded application payload When the data channel receives a MessagePack payload that does not match the reserved internal-signal shape.
camera Remote camera preview handle When an incoming track is classified as remote user-media and the remote camera preview handle is updated.
screen Remote screen preview handle When an incoming track is classified as remote display media and the remote screen preview handle is updated.

A binding MAY implement these notifications using DOM events, callbacks, listener registration methods, observer streams, channels, or any equivalent host-language event mechanism. The exported helper types P2PConnectionEventMap, P2PConnectionEventListener, and P2PConnectionEventListenerFor are one possible typing expression of this model and do not define additional runtime behavior.

Connection Abstraction

The central package abstraction is P2PConnection. A conforming binding MUST provide the following members and operations in host-language-appropriate form.

Member or operation Role Abstract behavior
makeOffer(additionalIceServers?) Association setup Creates local offer state and returns an Offer.
acceptOffer(offer, additionalIceServers?) Association setup Accepts an offer and returns paired contract copies.
Connection construction from Contract Construction Creates a live connection abstraction from one contract copy.
ready() Readiness Resolves when the connection's data channel becomes open.
shareMicrophone() Media publishing Publishes the shared local microphone track if available.
stopSharingMicrophone() Media publishing Stops publishing the current local microphone sender.
shareCamera() Media publishing Publishes the shared local camera track if available.
stopSharingCamera() Media publishing Stops publishing the current local camera sender.
shareScreen() Media publishing Publishes the shared local display stream if available.
stopSharingScreen() Media publishing Stops publishing current local display senders.
sendMessage(message) Application messaging Encodes and sends a structured application payload over the open data channel.
closeConnection() Shutdown Closes the live channel and the underlying peer connection.

The canonical JavaScript binding uses a constructor plus the method names shown above. Other bindings may expose the same semantics in a different syntactic form.

Common ICE Configuration

Both makeOffer() and acceptOffer() MUST create a new peer-connection object whose iceServers list begins with the following built-in STUN server record: 

{
  urls: [
    'stun:stun1.l.google.com:19302',
    'stun:stun2.l.google.com:19302',
    'stun:stun3.l.google.com:19302',
    'stun:stun4.l.google.com:19302',
  ],
}

If additionalIceServers is omitted, it MUST be treated as an empty list. When provided, its members MUST be appended in the order supplied by the caller. The created peer-connection object MUST use iceTransportPolicy: "all".

makeOffer()

  1. Create a new peer-connection object using the common ICE configuration.
  2. Create a data channel on that connection with the label "data".
  3. Generate an offerId using a host UUID facility.
  4. Create and set the local session description on the peer connection.
  5. Wait until ICE gathering has completed, either because the host peer-connection state becomes "complete" or because an ICE-candidate indication reports a null candidate.
  6. If the host peer connection does not expose a local description after that wait, throw MISSING_LOCAL_DESCRIPTION.
  7. Reserve the new peer connection and data channel in local pending state keyed by offerId.
  8. Return an Offer containing the offerId and finalized local description.

acceptOffer()

  1. Create a new peer-connection object using the common ICE configuration.
  2. Start waiting for the next incoming data-channel indication from that peer connection.
  3. Set the remote description from the supplied offer.description.
  4. Create and set a local session-description answer.
  5. Wait for ICE gathering completion using the same completion conditions defined for makeOffer().
  6. If the host peer connection does not expose a local description after that wait, throw MISSING_LOCAL_DESCRIPTION.
  7. Reserve the new peer connection and the pending incoming data channel promise in local accepted-offer state keyed by offer.offerId.
  8. Return a ContractCopies record whose offeror member contains the answer and whose offeree member identifies the offeree role.

Connection Construction

The connection-construction operation consumes exactly one reserved local state entry. A previously consumed contract MUST NOT be reusable.

  1. If the contract role is "offeror", look up the pending local offer state by contract.offerId.
  2. If no matching pending local offer exists, throw UNKNOWN_PEER_CONTRACT.
  3. Use the reserved peer connection and data channel as the live connection state, delete the pending reservation, and set the remote description from contract.answer.
  4. If the contract role is "offeree", look up the accepted local offer state by contract.offerId.
  5. If no matching accepted local offer exists, throw UNKNOWN_PEER_CONTRACT.
  6. Use the reserved peer connection and waiting incoming-channel promise as the live connection state, then delete the accepted-offer reservation.
  7. Mark an offeree connection as the polite peer and an offeror connection as the impolite peer for automatic renegotiation.
  8. When the live data channel becomes available, register a message handler that decodes MessagePack payloads as defined by [[MSGPACK]], consumes reserved internal signals, and dispatches all other payloads as application-message events.
  9. Register the automatic renegotiation and remote-track handlers defined later in this specification.

ready()

The ready() operation returns a promise-like readiness result that resolves only after the connection's live data channel has reached the "open" state.

The readiness result MUST resolve immediately if the channel is already open.

The readiness result MUST reject with:

  • CHANNEL_NOT_AVAILABLE if the peer connection reaches "failed" or "closed" before the offeree receives a data-channel indication;
  • CHANNEL_CLOSED if the channel closes before opening; or
  • CHANNEL_ERROR if the channel emits an error indication before opening.

sendMessage()

The sendMessage(message) operation MUST check whether the live data channel both exists and has readyState equal to "open".

If that condition is false, the operation MUST throw CONNECTION_NOT_READY. If it is true, the operation MUST MessagePack-encode the supplied value as defined by [[MSGPACK]] and send the resulting bytes on the live data channel.

The type parameter T constrains author intent in the canonical TypeScript binding, but the package does not perform runtime schema validation of decoded application payloads beyond internal-signal detection.

Listener Registration

A conforming binding MUST provide a way to register and remove listeners on the connection's own notification surface, not on the underlying peer-connection or data-channel primitives. The canonical JavaScript binding uses addEventListener() and removeEventListener().

closeConnection()

The closeConnection() operation MUST close the live data channel if one has been attached, and MUST then close the underlying peer connection.

Media Sharing and Preview Behavior

The package maintains shared local media state associated with the connection abstraction itself rather than with any single connection instance. That shared state is reused across all connections created in the same implementation context.

Shared state Meaning
localCameraVideoElement Shared local camera preview handle in a DOM-oriented binding.
localScreenVideoElement Shared local screen preview handle in a DOM-oriented binding.
Cached user-media stream Shared stream reused by shareMicrophone() and shareCamera().
Cached display-media stream Shared stream reused by shareScreen().

shareMicrophone()

  1. If no cached user-media stream exists, request one with both audio and video enabled and cache the resulting stream.
  2. Read the first audio track from that cached stream.
  3. If no audio track exists, return without throwing.
  4. Add that audio track to the peer connection and store the returned sender as the current user-audio sender.

shareCamera()

  1. If no cached user-media stream exists, request one with both audio and video enabled and cache the resulting stream.
  2. Read the first video track from that cached stream.
  3. If no video track exists, return without throwing.
  4. Add that video track to the peer connection and store the returned sender as the current user-video sender.
  5. If a DOM-oriented local camera preview handle does not exist, create one whose autoplay, inline-playback, and muted semantics are enabled.
  6. If the shared local camera preview handle has no source object, assign it a new stream containing the selected video track and begin playback.

shareScreen()

  1. If no cached display-media stream exists, request one with video and audio enabled and cache the resulting stream.
  2. Read the first video track from that cached stream. If present, add it to the peer connection and store the returned sender as the current display-video sender.
  3. Read the first audio track from that cached stream. If present, add it to the peer connection and store the returned sender as the current display-audio sender.
  4. If a DOM-oriented local screen preview handle does not exist, create one whose autoplay, inline-playback, and muted semantics are enabled.
  5. If the shared local screen preview handle has no source object, assign it a new stream containing the selected display video track and begin playback.

Stopping Shared Media

stopSharingMicrophone(), stopSharingCamera(), and stopSharingScreen() MUST remove the corresponding stored sender or senders from the peer connection when present and MUST do nothing when the corresponding sender is absent.

These operations do not stop the underlying cached media tracks, do not clear the cached streams, and do not remove or reset any shared preview handles.

Incoming Track Classification

When an incoming track indication is received, the package MUST use the first associated remote stream when available. If no stream is supplied, it MUST synthesize a new stream from the received track.

The first encountered remote stream identifier MUST be treated as the remote user-media stream. Tracks associated with that identifier update the remote camera preview handle and emit camera.

The next distinct remote stream identifier MUST be treated as the remote display stream. Tracks associated with that identifier update the remote screen preview handle and emit screen.

Automatic Renegotiation

Track additions may cause the underlying peer connection to report that renegotiation is needed. The package uses the established data channel as an internal renegotiation transport.

  1. If no live data channel has yet been attached to the connection, a renegotiation-needed indication MUST be ignored.
  2. Otherwise, the connection MUST create a fresh local offer, wait for ICE gathering completion, and send the resulting description across the data channel using the reserved internal-signal shape with __sovereignbase_peer2peer equal to "renegotiate-offer".
  3. When such an internal offer is received, the receiving connection MUST determine whether there is an offer collision.
  4. The offeror role acts as the impolite peer. The offeree role acts as the polite peer.
  5. If a colliding renegotiation offer is received by the impolite peer, that offer MUST be ignored.
  6. Otherwise, the receiving peer MUST apply the remote description. If the internal signal is a renegotiation offer, it MUST then create a fresh local answer, wait for ICE gathering completion, and send a "renegotiate-answer" internal signal back across the data channel.

Reserved internal signals MUST NOT be forwarded to application-message listeners.

Security and Privacy Considerations

Offer and answer values may contain ICE candidates and related network metadata from the host WebRTC stack. Applications should therefore treat their signaling transport as sensitive.

The package does not itself provide transport-layer confidentiality for signaling messages. Confidentiality and integrity for the peer-to-peer media and data path depend on the underlying WebRTC implementation.

Media capture operations rely on host permission prompts and permission policy. The package does not override or virtualize those consent requirements.
